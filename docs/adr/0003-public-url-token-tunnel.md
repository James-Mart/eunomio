# Public-URL sharing via Cloudflare quick tunnel, token-gated, on a second loopback listener

Eunomia binds `127.0.0.1` by design and has no application-level auth — the OS is the trust boundary. The "Public URL" feature on the home page lets the user open a Cloudflare quick tunnel from inside the tool so they can view sessions remotely (typically on a phone). This ADR records why that feature exists in the shape it does: a single button generates a URL containing a UUID share token, the token is checked by axum middleware on a second loopback listener that only exists while the tunnel is up, and the tunnel itself is supervised as a `cloudflared` child process inside the eunomia binary.

The original `share.sh` script used HTTP basic auth in a Caddy sidecar; we deliberately did not port that. Token-in-URL is the idiomatic shape for this category of tool (Jupyter and Gradio Share both use it for the same reason): one tap from a text message, no awkward basic-auth dialog on the phone, no user-chosen weak password, no `WWW-Authenticate` realm. The token is a UUID v4 (122 bits of entropy) generated fresh on every `POST /api/tunnel`; on the first hit, the middleware redirects to the same path with the query stripped and sets an `HttpOnly; Secure; SameSite=Lax` cookie so the token is no longer visible in the URL bar. To rotate, `DELETE` then `POST` — there is no separate "revoke" concept.

The auth listener is a second axum listener on `127.0.0.1:<ephemeral>`, only alive while a tunnel is running. We rejected three alternatives. Gating the main listener with the same middleware would force the local user to also present the token (bad UX) or require source-IP discrimination, which doesn't work because cloudflared connects from loopback too. Spawning a Caddy sidecar like `share.sh` does is the dev-mode arrangement, but inside the single-binary tool it adds a runtime dependency for no benefit. Cloudflare Access (named tunnel + Zero Trust) is the correct production-grade answer — IdP login, audit log, per-route policies — but it requires the user to own a domain on Cloudflare and configure a named tunnel up front, which kills the "click button, get link in five seconds" flow. Quick tunnel keeps that flow; we mark Cloudflare Access as a follow-up if anyone wants stronger auth.

Consequences worth knowing: the share token lives in memory only and is gone on restart (the trycloudflare URL rotates on restart anyway, so persistence buys nothing); the auth listener is bound to loopback so LAN hosts cannot bypass the token by hitting the port directly; the local `127.0.0.1:<port>` listener stays auth-free and is unchanged; `cloudflared` is auto-downloaded to `<data_dir>/bin/` for the linux/darwin/windows platform matrix the rest of the binary supports.

## Dev escape hatch

A hidden `--dev-tunnel` flag on the `eunomia` binary swaps the second listener for a direct cloudflared pointer at `http://localhost:5173` (the Vite dev server) and skips token generation entirely. This is set only by `npm run dev`'s backend invocation so HMR works end-to-end over the public URL; the trycloudflare subdomain itself is then the only secret. Release builds never set the flag, so the production tunnel design above is unaffected. The UI surfaces this state in `MobileShareCard` by hiding the token query param and showing a "Dev tunnel: no share token" warning.
