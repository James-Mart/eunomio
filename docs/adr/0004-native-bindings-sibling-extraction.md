# Native bindings ride alongside the SEA helper, extracted at runtime

The `cursor-helper` Node SEA executable embedded inside the eunomia binary bundles `@cursor/sdk` and our thin TypeScript shim into a single `node` runtime (see ADR 0001). The SDK's local runtime persists agent and run state through the `sqlite3` npm package, which is a JS wrapper around a `.node` C++ addon. Node SEAs can only embed JS into the SEA blob; native addons have to live on disk for `process.dlopen` to load them. So the helper bundle alone cannot satisfy `require("sqlite3")` — something has to put `node_sqlite3.node` somewhere `process.dlopen`-able and arrange for the JS to find it.

We ship the `.node` file as a sibling of the helper executable. `helper/build.mjs` copies `node_modules/sqlite3/build/Release/node_sqlite3.node` into `helper/dist/` after the SEA build, rust-embed picks the whole `dist/` tree up at `cargo build` time, and `cursor_bridge.rs::ensure_helper_extracted` writes both `cursor-helper` and every entry in `HELPER_NATIVE_FILES` into the same `/tmp/eunomia-helper-{version}/` directory on first use. The SDK's local runtime also needs a ripgrep binary: we copy `rg` from `@cursor/sdk-<platform>/bin/rg` into `helper/dist/` the same way and set `CURSOR_RIPGREP_PATH` to that absolute path when spawning `cursor-helper run`. esbuild aliases the `bindings` npm package to `helper/src/bindings-loader.cjs`, a dozen-line shim that resolves the requested `.node` against `path.dirname(process.execPath)` and `process.dlopen`s it. Adding another native dep is one entry in `HELPER_NATIVE_FILES` plus one entry in the build's copy list.

We considered three alternatives. Stubbing `sqlite3` to a throw-on-construct class — the shape we shipped at the partition checkpoint — was a deliberate deferral and explodes the moment the SDK's run-store opens. The SEA-native variant uses `sea-config.json`'s `assets` field plus `node:sea.getRawAsset` to embed the `.node` inside the SEA blob itself, which is conceptually tighter (single SEA file, no second artifact). We rejected it because `node:sea` only exists in Node ≥ 20.12, and the SEA helper inherits its runtime version from whichever `node` ran `build.mjs` — in environments like Cursor server's bundled `node` v20.11 that produces a binary that throws `ERR_UNKNOWN_BUILTIN_MODULE` at first dlopen. Trading a Node-version footgun for a marginal cleanliness win was not worth it. Replacing `sqlite3` with the built-in `node:sqlite` (since Node 22.5) or with an in-memory shim was rejected because the SDK pins to the `sqlite3` package by name and would need a fork or a deeper monkey-patch on every release.

Consequences worth knowing: the eunomia binary is already per-platform (the SEA helper is a copy of the build host's `node`), so adding per-platform `.node` files doesn't worsen the deploy story; rust-embed currently pulls the entire `helper/dist/` tree, which means intermediate build artifacts (`bundle.cjs`, `sea-prep.blob`) get embedded too — worth tightening with an `include` filter when binary size matters; SDK upgrades that change which native packages are pulled in have to be reflected in both `HELPER_NATIVE_FILES` and the copy list in `helper/build.mjs`, and there is no automatic check for drift. If the SDK ever picks up several more native deps, the pkg/nexe-style "extract a `node` runtime plus a real `node_modules` tree at runtime" pattern becomes the lower-friction follow-up.
